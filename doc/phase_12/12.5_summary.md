### 6. GARCH-Based Strategy
**File**: `crates/strategy/src/strategies/volatility/garch_strategy.rs`  
**Lines**: 614  
**Tests**: 11 (11 passing)  

**Hypothesis**: GARCH models predict volatility. Use predicted volatility for position timing - enter when predicted volatility is low (lower risk), exit when predicted volatility spikes.

**Key Features**:
- EWMA (Exponentially Weighted Moving Average) approximation of GARCH(1,1)
- Volatility prediction: `σ² = λ × σ²_{t-1} + (1-λ) × r²_{t-1}`
- Smoothing factor λ = 0.94 (typical for daily data)
- Volatility percentile-based entry (enter when vol < 40th percentile)
- Volatility spike exit (when predicted vol > 2× entry vol)

**Parameters**:
- `lambda`: 0.94
- `return_window`: 20
- `vol_entry_threshold`: 0.4
- `vol_exit_multiplier`: 2.0
- `fast_period`: 10
- `slow_period`: 30

## ⚠️ Challenges & Solutions

### Challenge 1: GARCH Complexity
**Issue**: Full GARCH(1,1) implementation is complex with many parameters

**Solution**: Simplified to EWMA (Exponentially Weighted Moving Average) approximation
- EWMA: `σ² = λ × σ²_{t-1} + (1-λ) × r²_{t-1}`
- λ = 0.94 (industry standard for daily data)
- Reduced from 7 parameters to 3
- Maintained core GARCH concept: volatility clustering

### Challenge 2: Crypto VIX Index
**Issue**: No official VIX for crypto markets

**Solution**: Constructed VIX-style index using ATR percentile ranking
- 0-100 scale matching traditional VIX
- Percentile-based (easier to interpret than raw volatility)
- Sentiment bands (Extreme Greed to Extreme Fear)
- Proven contrarian approach from traditional markets

### Challenge 3: Position Sizing in Spot-Only Model
**Issue**: Position sizing strategies don't naturally fit signal generation pattern

**Solution**: Encoded position size in signal strength field
- `strength = size_pct / 100.0` (normalized 0-1)
- Backtest engine can use strength for position sizing
- Maintain separation between signal generation and execution
- Documented clearly for users

### Challenge 4: Enum Variants
**Issue**: `MarketRegime` and `VolatilityLevel` enums lacked needed variants (`Transitioning`, `Variable`)

**Solution**: Used existing enum variants appropriately
- `Transitioning` → `HighVolatility` or `Ranging`
- `Variable` → `Medium` (representative)
- Maintains consistency with existing framework

## 📝 Assumptions & Decisions

### Documented Assumptions

1. **GARCH Simplification**: Full GARCH(1,1) approximated with EWMA for computational efficiency
   - **Justification**: EWMA captures volatility clustering core concept
   - **Impact**: Reduced complexity, faster execution, similar results
   
2. **Spot-Only Constraint**: All strategies are long-only (no short selling)
   - **Justification**: Project rules prohibit shorting
   - **Impact**: Limited to bullish strategies, contrarian approaches modified to exit-based
   
3. **Position Size in Signal**: Volatility-Adjusted Sizing encodes size in signal strength
   - **Justification**: Backtest engine expects signals, not separate sizing commands
   - **Impact**: Requires backtest engine to respect signal strength for position sizing
   
4. **Default Parameters**: All strategies use conservative default parameters
   - **Justification**: Prioritize risk management over maximum profit
   - **Impact**: Strategies may be less aggressive but more robust

### Design Decisions

1. **Modular Organization**: Created dedicated `volatility/` subdirectory
   - **Benefit**: Clear separation from other strategy categories
   - **Trade-off**: Additional level in directory hierarchy

2. **Configuration Pattern**: Used `Config` structs with validation for all strategies
   - **Benefit**: Type-safe configuration, compile-time validation
   - **Trade-off**: More boilerplate code per strategy

3. **Comprehensive Testing**: 10+ tests per strategy covering:
   - Creation and configuration
   - Metadata and traits
   - Core logic (entry/exit conditions)
   - Edge cases

4. **Hypothesis-First**: Each strategy documented with falsifiable claims
   - **Benefit**: Clear validation criteria, research-focused approach
   - **Trade-off**: Additional documentation effort
