Project Plan: Quantitative Crypto Trading System (Rust Edition)
Based on the "Unbiased Trading" Methodology
1. Executive Summary
This project aims to build a high-frequency capable, quantitative trading system for cryptocurrency markets using Rust. The core philosophy follows the "Unbiased Trading" approach: prioritizing statistical significance over "magic indicators," enforcing strict risk management, and validating strategies through rigorous, non-overfitted backtesting.
The system will feature a robust event-driven backtester, a time-series database for historical and live data, a safe execution engine, and a web-based dashboard for monitoring.
2. Core Philosophy & Methodology
To strictly adhere to the "Unbiased" approach, the system is designed to prevent common quant pitfalls.
2.1 The "Idea First" Rule
Constraint: Strategies cannot be generated by brute-forcing parameters on historical data.
Process: Every strategy must start with a written hypothesis (e.g., "Liquidations at 2 standard deviations create mean reversion opportunities") before any code is written.
Implementation: A "Hypothesis Register" (Markdown/DB) must be maintained.
2.2 Robustness over Raw Profit
Metric Focus: We prioritize SQN (System Quality Number) and Sharpe/Sortino Ratios over pure Net Profit.
Stress Testing: The backtester must include Monte Carlo simulations (shuffling trade order) to ensure the strategy survives bad luck sequences.
2.3 Survivorship Bias Free
Data Requirement: The database must include delisted coins to prevent testing only on "winners" (e.g., LUNA/FTX tokens must be in the historical set).
3. Technology Stack
3.1 Backend (Rust)
Core Logic: Rust (Stable). Selected for memory safety and zero-cost abstractions essential for low-latency execution.
Async Runtime: Tokio for handling concurrent websocket streams and database writes.
DataFrames: Polars for offline research and high-speed data manipulation (faster/safer than Pandas).
Math/Stats: ndarray for matrix operations and statrs for statistical distributions.
Web Server: Axum for the API and Dashboard backend.
3.2 Database
Primary: PostgreSQL with TimescaleDB extension.
Why: Best-in-class handling of time-series data (OHLCV, ticks) and strictly typed schemas.
ORM: sqlx (Rust) for compile-time checked SQL queries.
3.3 Frontend (Dashboard)
Framework: Vanilla JavaScript (served by Axum as static files).
Visualization: Plotly.js for candlestick charts, equity curves, and heatmaps.
4. System Architecture & Development Phases
Phase 1: Data Infrastructure ("The Source of Truth")
Goal: Build a robust data pipeline that guarantees data integrity.
1.1 Database Schema Design:
assets: Table for symbol details (tick size, lot size, status).
candles: Hypertable for OHLCV data. Crucial Field: ingested_at (to simulate latency).
trades: Hypertable for raw trade ticks (for high-fidelity backtests).
1.2 Ingestion Engine (Rust):
Implement binance-rs or barter-rs to fetch historical data (5+ years).
Build a "Gap Filler" service that identifies missing candles and auto-fetches them.
Deliverable: A populated TimescaleDB instance with 50+ pairs of clean data.
Phase 2: The "Unbiased" Backtesting Engine
Goal: Create a simulation environment that is harder to beat than the real market.
2.1 Event-Driven Architecture:
Avoid vectorized backtesting for final validation. Use an event loop that processes data candle-by-candle (or tick-by-tick).
Why: This prevents "Look-Ahead Bias" (using data from the future).
2.2 Simulation Fidelity:
Fee Model: Hardcode strict Taker/Maker fees (e.g., 0.1%).
Slippage Model: Dynamic slippage based on volatility (ATR).
Latency: Inject artificial delay (50ms-200ms) between Signal generation and Order fill.
2.3 Analysis Module:
Auto-generate reports: Max Drawdown Duration, Win Rate, Expectancy, SQN.
Walk-Forward Analysis: Train on Year 1, Test on Year 2. Roll forward.
Phase 3: Strategy Implementation
Goal: Modularize strategies so they can be hot-swapped.
3.1 The Strategy Trait:
Define a Rust Trait that requires on_candle, on_tick, and risk_check methods.
3.2 Initial Strategies (Unbiased Favorites):
Trend Following: Breakout with ATR trailing stop.
Mean Reversion: Bollinger Band deviations on lower timeframes.
Bias: Long-only on BTC > 200DMA, Cash otherwise.
Phase 4: Live Execution Engine (The "Black Box")
Goal: Safe, autonomous execution.
4.1 Signal-Executor Separation:
The Strategy calculates a "Target Position."
The Execution Engine calculates the difference between "Current" and "Target" and routes orders.
4.2 Risk Guardrails (Hardcoded):
Max Daily Loss: If equity drops > X%, flatten all positions and kill process.
Fat Finger Check: Reject orders > 5% of account balance unless explicitly authorized.
Drift Monitor: Alert if Live Fill Price deviates > 0.5% from Theoretical Price.
Phase 5: Dashboard & UI
Goal: Visualization and manual control.
5.1 API Layer (Axum):
Endpoints for account balance, open positions, and historical performance.
WebSocket for real-time log streaming.
5.2 React Interface:
Control Panel: "Start", "Stop", "Panic Close" buttons.
Live Charts: Plot strategy entry/exits over price data.
Console: Live scrolling log of Rust tracing events (Info/Warn/Error).
5. Implementation Roadmap (Staged Rollout)
Month 1: Foundation: Setup TimescaleDB, write Ingestion Engine, download History.
Month 2: The Lab: Build Backtester, implement standard metrics (Sharpe, Sortino), write first Strategy.
Month 3: Validation: Perform Walk-Forward and Monte Carlo tests. Refine strategy parameters (avoiding overfitting).
Month 4: Paper Trading: Run Live Engine connected to Binance Testnet. Compare Paper results vs Backtest model.
Month 5: Live Alpha: Deploy with minimal capital ($500-$1000). Monitor "Drift."
Month 6: Full Scale: Increase capital to target allocation.
6. Risk Management (The Unbiased Way)
Capital Allocation: Never risk more than 1-2% of account equity on a single trade setup.
Correlation Checks: If running multiple strategies, ensure they are not all Long BTC at the same time (use correlation matrices in ndarray).
Volatility Scaling: Reduce position size when ATR (Volatility) increases.
